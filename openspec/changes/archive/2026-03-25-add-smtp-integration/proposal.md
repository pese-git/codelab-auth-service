## Why

В текущей реализации сервис имеет заготовку для отправки email (файл `app/services/email_service.py`), но метод `send_confirmation_email` содержит только логирование без фактической отправки писем. Это препятствует полноценной работе функции подтверждения email при регистрации пользователей. Требуется реализовать фактическую отправку email через SMTP для:

- Отправки приветственных писем при регистрации
- Отправки писем подтверждения email адреса
- Информирования пользователя о важных событиях в системе

## What Changes

- **Реализация SMTP отправки:** Интеграция асинхронной библиотеки `aiosmtplib` для отправки email через SMTP сервер
- **Email шаблоны:** Создание системы шаблонов писем для разных типов уведомлений (приветствие, подтверждение, сброс пароля и т.д.)
- **Обработка ошибок:** Реализация retry логики при ошибках отправки, логирование неудачных попыток
- **Использование конфигурации:** Интеграция с существующими SMTP параметрами в `app/core/config.py` (smtp_host, smtp_port, smtp_username, smtp_password, smtp_from_email)
- **Расширение email_service:** Реализация метода `send_confirmation_email` с фактической отправкой писем
- **Интеграция в регистрацию:** Добавление отправки приветственного письма и письма подтверждения при регистрации пользователя в endpoint `POST /api/v1/register`
- **Асинхронное выполнение:** Опциональная отправка писем в background с использованием очередей (например, Redis) для непрерывной доступности API

## Capabilities

### New Capabilities

- `smtp-integration`: Фактическая отправка email через SMTP сервер с поддержкой аутентификации, TLS и обработкой ошибок
- `email-templates`: Система шаблонов для отправки разных типов email с поддержкой динамического контента
- `email-notifications`: Отправка email-уведомлений при событиях в системе (регистрация, подтверждение email)
- `email-retry-logic`: Механизм повтора отправки писем при временных ошибках соединения с SMTP

### Modified Capabilities

- `registration-flow`: Расширение flow регистрации с отправкой email-уведомлений при успешной регистрации
- `email-service`: Переход от заглушки к полной реализации email отправки в `EmailService`
- `configuration`: Использование существующих SMTP параметров и добавление новых при необходимости (например, максимальное количество retries)

## Impact

**Новые файлы:**
- `app/templates/emails/` - директория с HTML/текстовыми шаблонами писем
- `app/templates/emails/welcome.html` - шаблон приветственного письма
- `app/templates/emails/confirmation.html` - шаблон письма подтверждения email
- `app/services/email_sender.py` - класс для отправки email через SMTP (возможно отделение от email_service)

**Изменяемые файлы:**
- `app/services/email_service.py` - реализация методов `send_confirmation_email` и интеграция с SMTP
- `app/api/v1/register.py` - добавление отправки email при успешной регистрации
- `app/core/config.py` - возможное добавление параметров (smtp_max_retries, smtp_timeout, smtp_use_tls)
- `pyproject.toml` - добавление зависимостей `aiosmtplib` и `jinja2`
- `app/main.py` - инициализация SMTP клиента при запуске приложения (если используется connection pooling)

**Новые зависимости:**
- `aiosmtplib` - асинхронная библиотека для отправки email через SMTP
- `jinja2` - шаблонизатор для создания HTML писем

**Влияние на конфигурацию:**
- Требуется доступный SMTP сервер для отправки email (в dev можно использовать fake-smtp-server или MailHog)
- SMTP параметры должны быть установлены в переменных окружения или конфиге
- В production требуется использование проверенного SMTP провайдера (например, SendGrid, AWS SES, Mailgun)

**API изменения:**
- Endpoint `POST /api/v1/register` теперь отправляет email при успешной регистрации
- Response остаётся прежним, но в фоне происходит отправка email

**Безопасность:**
- Не логировать полные SMTP credentials в логах
- Обработать SMTP timeout и ошибки соединения без раскрытия деталей пользователю
- Использовать STARTTLS или SSL для шифрованной передачи по сети
