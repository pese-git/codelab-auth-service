# User Registration Спецификация

**Версия:** 1.0.0  
**Статус:** ✅ New Capability  
**Дата:** 2026-03-22

---

## 📋 Обзор

Новый публичный endpoint для самостоятельной регистрации пользователей с полной валидацией, безопасностью и аудит логированием.

---

## ADDED Requirements

### Requirement: Публичный endpoint регистрации
Система ДОЛЖНА предоставить публичный HTTP endpoint для регистрации новых пользователей без требования предварительной аутентификации.

#### Scenario: Успешная регистрация с валидными данными
- **WHEN** клиент отправляет POST запрос на `/api/v1/auth/register` с валидными email, username и паролем
- **THEN** система создаёт нового пользователя, возвращает 201 Created с данными пользователя (id, email, username, created_at)

#### Scenario: Регистрация с невалидным JSON
- **WHEN** клиент отправляет POST запрос с malformed JSON
- **THEN** система возвращает 400 Bad Request

#### Scenario: Регистрация без обязательных полей
- **WHEN** клиент отправляет POST запрос без одного из обязательных полей (email, username, password)
- **THEN** система возвращает 422 Unprocessable Entity с указанием недостающего поля

### Requirement: Валидация email адреса
Система ДОЛЖНА валидировать email адрес согласно RFC 5322 стандарту перед сохранением пользователя.

#### Scenario: Валидный email формат
- **WHEN** клиент регистрируется с email формата user@example.com
- **THEN** система принимает email и продолжает регистрацию

#### Scenario: Невалидный email формат
- **WHEN** клиент регистрируется с невалидным email (например, invalid@, @example.com, user @example.com)
- **THEN** система возвращает 422 Unprocessable Entity с сообщением об ошибке валидации email

#### Scenario: Email с поддежками
- **WHEN** клиент регистрируется с email содержащим плюс (user+tag@example.co.uk)
- **THEN** система принимает email как валидный

### Requirement: Валидация username
Система ДОЛЖНА валидировать username для обеспечения уникальности и корректного формата.

#### Scenario: Валидный username (3-20 символов)
- **WHEN** клиент регистрируется с username длиной 3-20 символов, содержащим буквы, цифры или символы -, _
- **THEN** система принимает username и продолжает регистрацию

#### Scenario: Слишком короткий username (< 3 символов)
- **WHEN** клиент регистрируется с username короче 3 символов
- **THEN** система возвращает 422 Unprocessable Entity

#### Scenario: Слишком длинный username (> 20 символов)
- **WHEN** клиент регистрируется с username длиннее 20 символов
- **THEN** система возвращает 422 Unprocessable Entity

#### Scenario: Username с недопустимыми символами
- **WHEN** клиент регистрируется с username содержащим спецсимволы типа !, @, #, $
- **THEN** система возвращает 422 Unprocessable Entity

### Requirement: Валидация пароля
Система ДОЛЖНА валидировать пароль на минимальную длину для обеспечения базовой безопасности.

#### Scenario: Валидный пароль (≥ 8 символов)
- **WHEN** клиент регистрируется с паролем длины 8 или более символов
- **THEN** система принимает пароль и продолжает регистрацию

#### Scenario: Слишком короткий пароль (< 8 символов)
- **WHEN** клиент регистрируется с паролем короче 8 символов
- **THEN** система возвращает 422 Unprocessable Entity с сообщением "Password must be at least 8 characters"

#### Scenario: Очень длинный пароль (> 128 символов)
- **WHEN** клиент регистрируется с паролем длиннее 128 символов
- **THEN** система возвращает 422 Unprocessable Entity

### Requirement: Проверка уникальности email
Система ДОЛЖНА гарантировать, что каждый email в системе уникален и предотвратить дублирование.

#### Scenario: Уникальный email
- **WHEN** клиент регистрируется с email который ещё не зарегистрирован
- **THEN** система создаёт пользователя успешно

#### Scenario: Email уже зарегистрирован
- **WHEN** клиент регистрируется с email который уже используется другим пользователем
- **THEN** система возвращает 409 Conflict с сообщением "Email already registered"

#### Scenario: Конкурентная регистрация с одинаковым email
- **WHEN** две request'ы одновременно пытаются зарегистрировать с одинаковым email
- **THEN** одна request успешна, другая получает 409 Conflict (защита database constraint)

### Requirement: Проверка уникальности username
Система ДОЛЖНА гарантировать, что каждый username в системе уникален и предотвратить дублирование.

#### Scenario: Уникальный username
- **WHEN** клиент регистрируется с username который ещё не используется
- **THEN** система создаёт пользователя успешно

#### Scenario: Username уже зарегистрирован
- **WHEN** клиент регистрируется с username который уже используется другим пользователем
- **THEN** система возвращает 409 Conflict с сообщением "Username already taken"

#### Scenario: Конкурентная регистрация с одинаковым username
- **WHEN** две request'ы одновременно пытаются зарегистрировать с одинаковым username
- **THEN** одна request успешна, другая получает 409 Conflict (защита database constraint)

### Requirement: Хеширование пароля
Система ДОЛЖНА хешировать пароль перед сохранением в БД используя bcrypt с cost factor 12.

#### Scenario: Пароль не сохраняется в plain text
- **WHEN** система создаёт пользователя с паролем
- **THEN** в БД сохраняется только bcrypt хеш пароля, не сам пароль

#### Scenario: Хеш пароля уникален для каждой регистрации
- **WHEN** два пользователя регистрируются с одинаковым паролем
- **THEN** их хеши паролей в БД различаются (благодаря random salt)

#### Scenario: Пароль не возвращается в response
- **WHEN** клиент успешно регистрируется
- **THEN** response содержит пользовательские данные (id, email, username) но НЕ содержит пароль или его хеш

### Requirement: Timing attack защита
Система ДОЛЖНА защищать от timing attacks которые позволяют узнать, существует ли email/username в системе.

#### Scenario: Одинаковое время ответа для существующего и несуществующего email
- **WHEN** клиент регистрируется с существующим email или с новым email
- **THEN** время ответа примерно одинаково (система выполняет полную валидацию и хеширование в обоих случаях)

#### Scenario: Аудит логирование при дубликате
- **WHEN** клиент пытается зарегистрировать существующий email
- **THEN** событие логируется в audit_logs даже при ошибке (не раскрывает информацию в response)

### Requirement: Аудит логирование попыток регистрации
Система ДОЛЖНА логировать все попытки регистрации (успешные и неудачные) в таблицу audit_logs.

#### Scenario: Успешная регистрация логируется
- **WHEN** пользователь успешно регистрируется
- **THEN** событие REGISTRATION_ATTEMPT_SUCCESS логируется с email, username, user_id, timestamp

#### Scenario: Неудачная регистрация логируется
- **WHEN** пользователь не проходит валидацию (слабый пароль, невалидный email и т.д.)
- **THEN** событие REGISTRATION_ATTEMPT_FAILED логируется с email, username, failure_reason, client_ip, timestamp

#### Scenario: Дублирование email логируется
- **WHEN** пользователь пытается зарегистрировать существующий email
- **THEN** событие REGISTRATION_DUPLICATE_EMAIL логируется с попытанным email, existing user_id

#### Scenario: Дублирование username логируется
- **WHEN** пользователь пытается зарегистрировать существующий username
- **THEN** событие REGISTRATION_DUPLICATE_USERNAME логируется с попытанным username, existing user_id

### Requirement: Response schema при успехе
Система ДОЛЖНА возвращать созданного пользователя с полной информацией при успешной регистрации.

#### Scenario: 201 Created response с пользовательскими данными
- **WHEN** регистрация успешна
- **THEN** система возвращает HTTP 201 Created с JSON телом содержащим id, email, username, created_at

#### Scenario: Response не содержит чувствительных данных
- **WHEN** регистрация успешна
- **THEN** response не содержит пароль, пароль хеш, refresh token или другие чувствительные данные

#### Scenario: Timestamp в ISO 8601 формате
- **WHEN** регистрация успешна
- **THEN** created_at в response находится в ISO 8601 формате (например, 2026-03-22T08:35:00.000Z)

### Requirement: Content-Type обработка
Система ДОЛЖНА корректно обрабатывать Content-Type header в request.

#### Scenario: application/json Content-Type
- **WHEN** клиент отправляет POST запрос с Content-Type: application/json
- **THEN** система корректно парсит JSON тело запроса

#### Scenario: Неверный Content-Type
- **WHEN** клиент отправляет POST запрос с Content-Type: text/plain или другой неправильный тип
- **THEN** система возвращает 415 Unsupported Media Type (или обрабатывает согласно FastAPI default поведению)

### Requirement: CORS поддержка
Система ДОЛЖНА поддерживать CORS для публичного endpoint регистрации.

#### Scenario: CORS preflight request
- **WHEN** браузер отправляет OPTIONS запрос на /api/v1/register
- **THEN** система возвращает 200 OK с корректными CORS headers

#### Scenario: Кросс-доменная регистрация
- **WHEN** браузер на другом домене отправляет POST регистрацию
- **THEN** система позволяет запрос согласно CORS политике

### Requirement: Email confirmation (опциональное)
Система ДОЛЖНА поддерживать опциональное подтверждение email адреса при регистрации через environment переменную для гибкости development и production окружений.

#### Scenario: Email confirmation включено (REQUIRE_EMAIL_CONFIRMATION=true)
- **WHEN** клиент регистрируется с email confirmation включено
- **THEN** пользователь создается в статусе `unconfirmed`, система возвращает 201 Created с information что требуется подтверждение

#### Scenario: Email confirmation отключено (REQUIRE_EMAIL_CONFIRMATION=false)
- **WHEN** клиент регистрируется с email confirmation отключено
- **THEN** пользователь создается в статусе `active` и может сразу использовать систему

#### Scenario: Confirmation email отправляется
- **WHEN** email confirmation включено и пользователь регистрируется
- **THEN** система отправляет email с confirmation ссылкой на регистрационный email адрес

#### Scenario: Пользователь может подтвердить email
- **WHEN** пользователь с unconfirmed статусом получает email и переходит по ссылке подтверждения
- **THEN** система обновляет статус на `confirmed` и пользователь может полностью использовать систему

#### Scenario: Неподтвержденный пользователь не может логиниться
- **WHEN** пользователь с unconfirmed статусом пытается логиниться через /oauth/token
- **THEN** система может возвращать ошибку или требовать подтверждение (в зависимости от бизнес логики)

## 📋 Дополнительные требования

### Email confirmation конфигурируется через переменные окружения
- `REQUIRE_EMAIL_CONFIRMATION` (default: true) — включить/отключить требование подтверждения email
- Переменная позволяет использовать разные стратегии для development (часто отключено) и production (обычно включено)

### Нет автоматического логирования после регистрации
- После успешной регистрации пользователь НЕ автоматически логируется
- Пользователь должен использовать `/oauth/token` endpoint для получения access/refresh токенов

## 📊 HTTP Статус Коды

| Код | Значение | Сценарий |
|-----|----------|---------|
| **201** | Created | Успешная регистрация |
| **400** | Bad Request | Malformed JSON, отсутствуют поля |
| **409** | Conflict | Дублирование email или username |
| **415** | Unsupported Media Type | Неверный Content-Type |
| **422** | Unprocessable Entity | Невалидные данные (email формат, пароль слабый и т.д.) |
| **429** | Too Many Requests | Rate limit превышен |
