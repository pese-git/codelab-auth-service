# Модель токенов сброса пароля Спецификация

**Версия:** 1.0.0  
**Статус:** ✅ New Capability  
**Дата:** 2026-03-24

---

## 📋 Обзор

Создание модели базы данных и логики управления одноразовыми токенами сброса пароля с криптографической защитой, сроком действия и механизмом предотвращения повторного использования.

---

## ADDED Requirements

### Requirement: Модель токена сброса пароля

Система ДОЛЖНА хранить в базе данных криптографически защищённые одноразовые токены для восстановления пароля с временем истечения.

#### Scenario: Создание токена сброса
- **WHEN** пользователь запрашивает сброс пароля
- **THEN** система генерирует уникальный токен (32 символа, secrets.token_urlsafe), хранит хеш (SHA-256) в таблице password_reset_tokens с user_id, временем создания и временем истечения (30 минут)

#### Scenario: Верификация токена
- **WHEN** пользователь предоставляет токен для подтверждения сброса пароля
- **THEN** система проверяет что токен существует, не истёк и не был использован ранее

#### Scenario: Истечение срока действия токена
- **WHEN** прошло 30 минут с момента создания токена
- **THEN** система отвергает любую попытку сброса пароля с использованием этого токена

#### Scenario: Одноразовое использование
- **WHEN** токен успешно использован для сброса пароля
- **THEN** система отмечает токен как использованный (used_at = now) и запрещает повторное использование того же токена

### Requirement: Безопасное хранение токенов

Система ДОЛЖНА хранить токены в хешированном виде в базе данных для предотвращения несанкционированного сброса пароля при компрометации базы данных.

#### Scenario: Хеширование токена при сохранении
- **WHEN** токен сброса пароля сохраняется в базу данных
- **THEN** токен хешируется с использованием SHA-256 перед сохранением, в БД хранится только хеш

#### Scenario: Сравнение токена с хешем
- **WHEN** пользователь предоставляет токен для верификации
- **THEN** система хеширует предоставленный токен и сравнивает с сохранённым хешем, используя constant-time сравнение

### Requirement: Генерация криптографически стойких токенов

Система ДОЛЖНА генерировать криптографически надёжные случайные токены, непредсказуемые для атакующего.

#### Scenario: Генерация случайного токена
- **WHEN** создаётся новый токен сброса пароля
- **THEN** система генерирует 32-символьный случайный токен используя secrets.token_urlsafe() (рекомендуется минимум 24 байта энтропии)

#### Scenario: Уникальность токена
- **WHEN** проверяется новогенерированный токен на наличие в БД
- **THEN** все сгенерированные токены уникальны и не совпадают с существующими

### Requirement: Таблица базы данных password_reset_tokens

Система ДОЛЖНА содержать таблицу для хранения информации о токенах сброса пароля.

#### Scenario: Структура таблицы password_reset_tokens
- **WHEN** приложение инициализируется
- **THEN** таблица password_reset_tokens содержит колонки: id (UUID primary key), user_id (FK на users), token_hash (VARCHAR, unique), created_at (TIMESTAMP, UTC), expires_at (TIMESTAMP, UTC), used_at (TIMESTAMP nullable, UTC)

#### Scenario: Индексы на таблице
- **WHEN** таблица password_reset_tokens создана
- **THEN** существуют индексы на: user_id (для быстрого поиска токенов пользователя), expires_at (для очистки истёкших токенов), token_hash (для верификации)

## Constraints

- **Криптография**: Использование secrets.token_urlsafe для генерации и SHA-256 для хеширования
- **Срок действия**: Токены действуют ровно 30 минут, после чего отклоняются
- **Одноразовость**: Каждый токен может быть использован максимум один раз
- **Async**: Все операции с БД должны быть асинхронными (async/await)
- **Безопасность логирования**: Токены и их хеши НЕ должны логироваться

---

## Dependencies

- **Зависит от**: `user-authentication` (модель User и foreign key)
- **Используется в**: `password-reset-api` (верификация и использование токенов)
- **Используется в**: `password-reset-notifications` (генерация при запросе сброса)

## Testing

### Unit Tests

1. **test_token_generation**
   - Проверить что сгенерированный токен имеет правильный формат и содержит достаточно энтропии

2. **test_token_hashing**
   - Проверить что хеш одного и того же токена всегда идентичен
   - Проверить что reverse хеширование невозможно

3. **test_token_expiration**
   - Проверить что истёкший токен отвергается
   - Проверить что валидный токен принимается

4. **test_single_use_enforcement**
   - Проверить что второе использование одного токена отвергается

### Integration Tests

1. **test_token_lifecycle**
   - Создать токен → Верифицировать → Использовать → Попытка повторного использования (должна провалиться)
