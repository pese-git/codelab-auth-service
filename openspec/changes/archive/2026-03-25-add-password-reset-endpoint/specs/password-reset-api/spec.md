# REST API endpoints сброса пароля Спецификация

**Версия:** 1.0.0  
**Статус:** ✅ New Capability  
**Дата:** 2026-03-24

---

## 📋 Обзор

REST API endpoints для запроса восстановления пароля и подтверждения сброса пароля с установкой нового пароля, включая валидацию, обработку ошибок и защиту от brute-force атак.

---

## ADDED Requirements

### Requirement: Endpoint запроса сброса пароля

Система ДОЛЖНА предоставить POST endpoint для инициирования процесса сброса пароля по email адресу пользователя.

#### Scenario: Успешный запрос сброса
- **WHEN** пользователь отправляет POST запрос на `/api/v1/auth/password-reset/request` с корректным email
- **THEN** система проверяет наличие пользователя с этим email, генерирует токен, отправляет письмо и возвращает статус 200 с сообщением «Инструкции по восстановлению пароля отправлены»

#### Scenario: Несуществующий email
- **WHEN** пользователь отправляет запрос с email, под который не зарегистрирована учётная запись
- **THEN** система возвращает статус 200 с тем же сообщением (не раскрывает информацию о существовании аккаунта)

#### Scenario: Невалидный email формат
- **WHEN** пользователь отправляет запрос с некорректным email адресом
- **THEN** система возвращает статус 400 с ошибкой валидации

#### Scenario: Rate limiting на запросы
- **WHEN** пользователь отправляет более 3 запросов на восстановление пароля в течение 1 часа
- **THEN** система возвращает статус 429 (Too Many Requests) и блокирует дальнейшие запросы на 1 час

### Requirement: Endpoint подтверждения сброса пароля

Система ДОЛЖНА предоставить POST endpoint для подтверждения сброса пароля и установки нового пароля.

#### Scenario: Успешное подтверждение и смена пароля
- **WHEN** пользователь отправляет POST запрос на `/api/v1/auth/password-reset/confirm` с корректным токеном и новым пароль
- **THEN** система верифицирует токен, валидирует новый пароль, обновляет пароль пользователя и возвращает статус 200

#### Scenario: Невалидный токен
- **WHEN** пользователь предоставляет несуществующий или невалидный токен
- **THEN** система возвращает статус 400 с ошибкой «Ссылка для восстановления недействительна»

#### Scenario: Истёкший токен
- **WHEN** пользователь предоставляет токен, срок действия которого истёк (более 30 минут)
- **THEN** система возвращает статус 400 с ошибкой «Ссылка для восстановления истекла, запросите новую»

#### Scenario: Повторное использование токена
- **WHEN** пользователь пытается использовать уже использованный токен
- **THEN** система возвращает статус 400 с ошибкой «Ссылка уже была использована»

#### Scenario: Невалидный новый пароль
- **WHEN** пользователь отправляет пароль, не соответствующий требованиям (менее 8 символов или отсутствуют заглавные буквы)
- **THEN** система возвращает статус 400 с описанием требований к паролю

#### Scenario: Успешное изменение пароля
- **WHEN** пароль успешно изменён
- **THEN** система отмечает токен как использованный, логирует событие смены пароля и возвращает статус 200

### Requirement: Защита от brute-force атак

Система ДОЛЖНА реализовать механизм защиты от перебора токенов и попыток угадания кода.

#### Scenario: Ограничение попыток подтверждения
- **WHEN** система получает более 10 неудачных попыток подтверждения с разными токенами из одного IP адреса в течение 5 минут
- **THEN** система блокирует этот IP адрес на 15 минут

#### Scenario: Логирование подозрительной активности
- **WHEN** обнаруживается pattern brute-force попыток (много неудачных попыток из одного IP)
- **THEN** система логирует событие как security warning с IP адресом и timestamp

### Requirement: Обработка ошибок и безопасность

Система ДОЛЖНА корректно обрабатывать ошибки и не раскрывать чувствительную информацию.

#### Scenario: CORS и аутентификация
- **WHEN** запрос приходит с другого домена
- **THEN** система проверяет CORS политику и возвращает 403 если домен не в whitelist

#### Scenario: Логирование без чувствительных данных
- **WHEN** происходит ошибка в процессе сброса пароля
- **THEN** логируется событие без пароля, токена и деталей pii

#### Scenario: HTTPS требование
- **WHEN** запрос приходит через HTTP вместо HTTPS
- **THEN** система перенаправляет на HTTPS версию (в production)

## Data Structures

### Request: Password Reset Request

```json
{
  "email": "user@example.com"
}
```

**Constraints**:
- email: непустой, валидный email адрес (RFC 5322)

### Request: Password Reset Confirm

```json
{
  "token": "abc123_def456_ghi789...",
  "password": "NewP@ssw0rd123",
  "password_confirm": "NewP@ssw0rd123"
}
```

**Constraints**:
- token: непустая строка, формат token_urlsafe
- password: 8-64 символа, содержит заглавные, строчные буквы, цифры и специальные символы
- password_confirm: должен совпадать с password

### Response: Success (200)

```json
{
  "status": "success",
  "message": "Пароль успешно изменён"
}
```

### Response: Error (400/429)

```json
{
  "status": "error",
  "message": "Описание ошибки",
  "code": "error_code"
}
```

## Constraints

- **Rate Limiting**: Максимум 3 запроса на сброс в час на пользователя, максимум 10 попыток подтверждения на IP в 5 минут
- **Токен действует**: Ровно 30 минут
- **Пароль требования**: минимум 8 символов, должен содержать: заглавные, строчные буквы, цифры, спецсимволы
- **Безопасность**: Токены и пароли НЕ логируются в полном виде, используется HTTPS в production
- **Async**: Все операции должны быть асинхронными
