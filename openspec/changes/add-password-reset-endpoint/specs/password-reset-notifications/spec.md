# Email уведомления сброса пароля Спецификация

**Версия:** 1.0.0  
**Статус:** ✅ New Capability  
**Дата:** 2026-03-24

---

## 📋 Обзор

Email уведомления пользователю при запросе сброса пароля с безопасной ссылкой для восстановления доступа, отправляемые через существующую SMTP инфраструктуру.

---

## ADDED Requirements

### Requirement: Отправка письма с ссылкой сброса пароля

Система ДОЛЖНА отправлять email пользователю при запросе восстановления пароля с безопасной ссылкой и инструкциями.

#### Scenario: Письмо отправлено после запроса сброса
- **WHEN** пользователь запрашивает сброс пароля, система генерирует токен
- **THEN** система отправляет email на адрес пользователя с содержимым: тема письма, приветствие, ссылка с токеном, instrукции, срок действия (30 мин), link в формате `https://domain.com/auth/reset-password?token={token}`

#### Scenario: Корректное содержимое письма
- **WHEN** письмо сброса пароля отправляется
- **THEN** письмо содержит: пользовательское имя, ссылку для сброса, информацию о сроке действия, warning что ссылка действует только 30 минут, и опцию игнорирования если не запрашивал

#### Scenario: HTML и текстовая версии письма
- **WHEN** письмо отправляется
- **THEN** система отправляет письмо как multipart/alternative с HTML и текстовой версиями для совместимости

#### Scenario: Письмо отправляется асинхронно
- **WHEN** пользователь отправляет запрос на сброс пароля
- **THEN** endpoint возвращает 200 ответ сразу, письмо отправляется в background без блокирования запроса

#### Scenario: Безопасная ссылка в письме
- **WHEN** письмо содержит ссылку сброса пароля
- **THEN** ссылка включает: безопасный токен, не содержит другие sensitive данные (пароль, username), использует HTTPS для production

### Requirement: Обработка ошибок отправки

Система ДОЛЖНА надёжно обрабатывать ошибки отправки email и убедить пользователя повторить попытку при необходимости.

#### Scenario: Ошибка отправки email
- **WHEN** email не может быть отправлен по какой-либо причине (SMTP ошибка, таймаут)
- **THEN** токен остаётся активным в БД, пользователь может повторить запрос, ошибка логируется без раскрытия деталей пользователю

#### Scenario: Retry логика для отправки
- **WHEN** при отправке письма происходит временная ошибка (network timeout, SMTP timeout)
- **THEN** система повторяет отправку до 3 раз с экспоненциальным backoff (1сек, 2сек, 4сек) прежде чем отказаться

#### Scenario: Успешное логирование отправки
- **WHEN** письмо успешно отправляется
- **THEN** событие логируется с информацией: recipient email (masked), message_id, timestamp, без логирования токена

### Requirement: Защита от spam и abuse

Система ДОЛЖНА реализовать механизм защиты от использования email уведомлений для spam рассылок.

#### Scenario: Отправка письма только для существующих пользователей
- **WHEN** запрос на сброс пароля приходит
- **THEN** письмо отправляется ТОЛЬКО если в системе существует пользователь с таким email, но в response не раскрывается существует ли такой пользователь

#### Scenario: Одно письмо на запрос
- **WHEN** пользователь отправляет запрос на сброс пароля
- **THEN** отправляется ровно одно письмо, при повторном запросе отправляется новое письмо (не дублирование)

#### Scenario: Предотвращение spam список
- **WHEN** много писем попадает в spam (bounce rate > 5% за неделю)
- **THEN** система снижает отправку таким пользователям или требует проверки email

### Requirement: Email шаблон для сброса пароля

Система ДОЛЖНА использовать подготовленный шаблон email для отправки уведомлений о сбросе пароля.

#### Scenario: Загрузка шаблона
- **WHEN** инициируется отправка email сброса пароля
- **THEN** система загружает шаблон из `app/templates/emails/password_reset/` (subject.txt и template.html, как для других email типов)

#### Scenario: Подстановка переменных в шаблон
- **WHEN** шаблон готовится для отправки
- **THEN** переменные подставляются: user_name, reset_url, token_expiry_hours, support_email, current_year, app_name

#### Scenario: Локализация шаблона
- **WHEN** шаблон загружается
- **THEN** содержимое может быть локализовано на языке пользователя (если информация доступна в профиле, иначе используется язык по умолчанию)

## Data Structures

### Email Template Variables

```python
{
    "user_name": str,  # Имя пользователя
    "reset_url": str,  # Полный URL с токеном: https://domain/auth/reset-password?token=...
    "token_expiry_hours": int,  # Часы до истечения (обычно 0.5 = 30 минут)
    "support_email": str,  # Email для поддержки
    "current_year": int,  # Текущий год для footer
    "app_name": str,  # Название приложения
}
```

## Constraints

- **Email доставка**: Письма должны доставляться в течение 5 минут (или уведомлять пользователя об ошибке)
- **Async**: Отправка писем должна быть асинчронной, не блокировать API запрос
- **Безопасность**: Токены и пароли НЕ должны логироваться, sensible данные в письмах минимизированы
- **Retry**: Максимум 3 попытки отправки с exponential backoff
- **SMTP**: Использование существующей SMTP интеграции из `email-notifications` capability
- **Шаблоны**: Использование существующей системы шаблонов из `email-templates` capability

## Dependencies

- **Зависит от**: `smtp-integration` (фактическая отправка email через SMTP)
- **Зависит от**: `email-templates` (система шаблонов)
- **Зависит от**: `email-notifications` (общая логика отправки уведомлений)
