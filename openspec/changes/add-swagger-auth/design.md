# Дизайн интеграции OAuth2 Bearer в Swagger UI

## Context

В настоящее время FastAPI приложение предоставляет API с OAuth2 аутентификацией, однако Swagger UI (доступный по адресу `/docs`) не поддерживает интерактивное тестирование защищённых эндпоинтов с использованием Bearer токенов. Разработчики и тестировщики вынуждены использовать внешние инструменты (curl, Postman) для работы с защищёнными эндпоинтами, что замедляет разработку и тестирование.

Существующая система аутентификации использует JWT-based access tokens (RS256), которые выдаются через эндпоинт `/oauth/token`. Текущая Swagger документация не использует встроенные возможности FastAPI для отображения этой функциональности.

## Goals / Non-Goals

**Goals:**
- Добавить OAuth2 Bearer security scheme в конфигурацию FastAPI для Swagger UI
- Отобразить кнопку "Authorize" в Swagger UI для ввода AccessToken
- Автоматически передавать введённый токен в заголовке `Authorization: Bearer <token>` при тестировании эндпоинтов
- Визуально пометить все защищённые эндпоинты в документации с иконкой замка
- Обеспечить совместимость с существующей JWT-based системой аутентификации
- Использовать переменную окружения для управления доступностью Swagger UI в разных окружениях

**Non-Goals:**
- Добавление новых типов аутентификации или новых систем авторизации
- Изменение логики валидации токенов или существующих зависимостей аутентификации
- Добавление графического интерфейса для создания токенов в Swagger
- Поддержка других OAuth2 flows (authorization_code, client_credentials и т.д.) в Swagger
- Изменение структуры токенов или JWT claims
- Логирование попыток авторизации в Swagger для аудита

## Decisions

### Decision 1: Использование встроенного HTTPBearer в FastAPI

**Choice**: Использовать встроенный класс `HTTPBearer` из FastAPI для определения security scheme в Swagger.

**Rationale**: 
- FastAPI имеет встроенную поддержку OAuth2 schemes, которая автоматически генерирует правильную OpenAPI/Swagger конфигурацию
- Это решение не требует дополнительных зависимостей
- Совместимо с существующей JWT-based системой
- Позволяет использовать встроенные механизмы FastAPI для документирования защищённых эндпоинтов

**Alternative Considered**: 
- Вручную редактировать OpenAPI schema — слишком сложно и подвержено ошибкам
- Использовать сторонние библиотеки — избыточно, так как FastAPI имеет встроенную поддержку

### Decision 2: Централизованное определение security scheme в app/main.py

**Choice**: Определить security scheme один раз в главном приложении FastAPI в файле `app/main.py` с использованием встроенного механизма OpenAPI.

**Rationale**:
- Централизованное определение упрощает обслуживание и позволяет управлять схемой безопасности в одном месте
- Позволяет применить security scheme ко всем приложениям (главное приложение)
- Избегает дублирования конфигурации в разных файлах

**Alternative Considered**:
- Определение в отдельном файле конфигурации — усложняет логику инициализации приложения
- Определение на уровне роутеров — более гибко, но сложнее для управления

### Decision 3: Использование HTTPBearer для Bearer token аутентификации

**Choice**: Использовать `HTTPBearer` из FastAPI для более точного представления схемы Bearer token.

**Rationale**:
- `HTTPBearer` лучше соответствует реальной схеме аутентификации (Bearer token в заголовке Authorization)
- Генерирует правильную OpenAPI схему `type: "http"` с `scheme: "bearer"`
- Не требует специальных полей (как password и username в OAuth2PasswordBearer)

**Alternative Considered**:
- OAuth2PasswordBearer — имеет дополнительные поля и flows, которые не применимы к Bearer токенам

### Decision 4: Интеграция с существующей функцией get_current_user

**Choice**: Новая Swagger конфигурация работает вместе с существующей функцией `get_current_user` в `app/core/dependencies.py` без модификации.

**Rationale**:
- Существующая функция уже валидирует Bearer tokens и извлекает пользователя
- Swagger поддержка — это только документирование и удобство тестирования
- Не требуется изменение логики авторизации

**Alternative Considered**:
- Создание новой функции для Swagger — усложняет код и дублирует логику

### Decision 5: Управление доступностью Swagger UI через переменную окружения

**Choice**: Использовать переменную окружения `ENABLE_SWAGGER_UI` для управления доступностью Swagger UI в разных окружениях.

**Rationale**:
- В production Swagger UI может содержать чувствительную информацию о структуре API
- Переменная окружения позволяет гибко отключать/включать документацию без изменения кода
- Стандартный подход для управления функциональностью в разных окружениях

**Implementation**:
- Если `ENABLE_SWAGGER_UI=false` (или отсутствует в production), параметры `docs_url=None` и `redoc_url=None` передаются в конструктор FastAPI
- Если `ENABLE_SWAGGER_UI=true`, Swagger UI доступен по адресу `/docs`

## Risks / Trade-offs

**Risk 1: Bearer токен виден в Swagger UI**
→ **Mitigation**: Это работает только в локальной среде разработки. В production Swagger UI отключается через переменную окружения. Пользователи должны использовать разные токены для разных окружений.

**Risk 2: Неправильная валидация в Swagger может скрыть реальные ошибки**
→ **Mitigation**: Тестирование в Swagger не заменяет автоматизированные тесты. Разработчики должны запускать интеграционные тесты перед комитом.

**Risk 3: Потенциальный конфликт с существующими security definitions**
→ **Mitigation**: Предварительно проверить, что в конфигурации FastAPI нет других security schemes, которые могут конфликтовать.

**Trade-off 1: Простота vs. гибкость**
→ Выбираем простоту: используем встроенные возможности FastAPI вместо кастомной реализации. Это ограничивает гибкость, но упрощает поддержку.

**Trade-off 2: Локальное тестирование vs. безопасность**
→ Выбираем оба: Swagger UI включён по умолчанию в dev для удобства, но может быть отключён в production через переменную окружения.

## Migration Plan

1. **Этап 1 — Конфигурация (non-breaking)**
   - Добавить `HTTPBearer` конфигурацию в `app/main.py`
   - Добавить переменную окружения `ENABLE_SWAGGER_UI` с значением по умолчанию `true`
   - Проверить, что документация генерируется правильно

2. **Этап 2 — Валидация**
   - Проверить в Swagger UI, что кнопка "Authorize" отображается
   - Проверить, что введённый токен отправляется в заголовке Authorization
   - Протестировать с корректным и некорректным токеном
   - Проверить, что Swagger отключается когда `ENABLE_SWAGGER_UI=false`

3. **Этап 3 — Откат (если потребуется)**
   - Удалить конфигурацию HTTPBearer из main.py
   - Удалить переменную окружения `ENABLE_SWAGGER_UI`
   - Перезагрузить приложение
   - Swagger вернётся к состоянию без авторизации

Нет требований к миграции данных или обновления БД, так как это только изменение в документации API.

## Implementation Notes

- Переменная окружения `ENABLE_SWAGGER_UI` должна быть добавлена в файл конфигурации `app/core/config.py`
- Значение по умолчанию: `ENABLE_SWAGGER_UI=true` (для разработки)
- В production следует установить `ENABLE_SWAGGER_UI=false` в `.env.production`
