## Контекст

В системе аутентификации отсутствует публичный механизм регистрации пользователей. Пользователи могут быть созданы только во время инициализации базы данных через seed скрипты. Это препятствует работе приложения в production окружении, где требуется возможность самостоятельной регистрации пользователей.

В системе уже реализована инфраструктура для:
- Модели User с ограничениями уникальности на email и username
- Хеширования паролей с помощью bcrypt (cost factor 12)
- Rate limiting middleware для защиты endpoint'ов от abuse
- Аудита логирования всех событий безопасности
- Валидации входных данных через Pydantic schemas
- REST API на основе FastAPI с версионированием v1

## Цели / Не-цели

**Цели:**
- Предоставить безопасный публичный endpoint `/api/v1/register` для самостоятельной регистрации пользователей
- Валидировать данные регистрации (email, username, пароль) перед сохранением
- Предотвратить дублирование email/username через проверку уникальности
- Защитить endpoint от abuse через rate limiting (ограничение по IP адресу)
- Логировать все попытки регистрации (успешные и ошибки) для аудита
- Возвращать информативные сообщения об ошибках валидации
- Следовать существующим паттернам и практикам безопасности системы
- Включить самостоятельную регистрацию пользователей в production

**Не-цели:**
- Социальные сети логины (OAuth, Google и т.д.)
- Сложные требования к паролям - только минимальная длина в текущей версии
- Заполнение профиля после регистрации
- Многоязычные сообщения об ошибках

## Решения

### 1. Дизайн endpoint'а: публичный доступ без аутентификации
**Решение:** Endpoint `POST /api/v1/register` будет полностью публичным без требования аутентификации.

**Обоснование:**
- Регистрация должна быть доступна новым пользователям без учетных данных
- Самостоятельная регистрация - основное требование для production готовности
- Стандартный REST API паттерн для публичных endpoint'ов регистрации

**Рассмотренные альтернативы:**
- **С аутентификацией:** Требовал бы отдельного admin API, ограничил бы самостоятельность
- **Только по приглашениям:** Ограничивает рост, требует участия admin'а для каждой регистрации

### 2. Безопасность пароля: Bcrypt с cost factor 12
**Решение:** Использовать существующую реализацию bcrypt с cost factor 12 через Pydantic валидацию.

**Обоснование:**
- Уже реализовано в codebase через `crypto.py` утилиты
- Cost factor 12 обеспечивает ~256ms хеширования на современном железе - достаточно для безопасности
- Bcrypt с генерацией salt предотвращает rainbow table атаки
- Согласуется с существующей обработкой паролей в системе

**Рассмотренные альтернативы:**
- **Argon2:** Лучшие характеристики безопасности, но не в текущих зависимостях
- **PBKDF2:** Широко поддерживается, но менее устойчив к GPU атакам чем bcrypt
- **Cost factor 10 или выше:** 10 быстрее но слабее; 13+ добавит ненужную задержку

### 3. Валидация входных данных: отдельная схема для регистрации
**Решение:** Создать новую Pydantic схему `UserRegister` отдельно от основной схемы `User`.

**Обоснование:**
- Разделение ответственности - input регистрации != сохраненное представление пользователя
- Позволяет строгую валидацию пароля без влияния на другие схемы
- Обеспечивает ясный контракт API для request'ов регистрации

**Структура:**
```
UserRegister (request schema):
  - email: str (валидируется как email)
  - username: str (буквы/цифры, 3-20 символов, проверка уникальности)
  - password: str (минимум 8 символов, хешируется перед сохранением)

User (response schema):
  - id, email, username, created_at (без поля password)
```

### 4. Предотвращение дубликатов: проверка на уровне приложения + ограничения БД
**Решение:** Проверять уникальность на уровне приложения перед INSERT, полагаться на database constraint как на страховку.

**Обоснование:**
- **Проверка приложения:** Обеспечивает немедленный, специфичный error response для лучшего UX
- **Database constraint:** Обрабатывает race conditions где две concurrent request'ы пытаются вставить дубликат
- Двухуровневая защита предотвращает 500 errors у клиентов

**Процесс:**
1. Query существующих пользователей по email и username
2. Если найдены, return 409 Conflict с field-specific ошибкой
3. Database constraint обеспечивает отсутствие data corruption если app проверка пропущена

### 5. Rate limiting: по IP адресу, интеграция с существующим middleware
**Решение:** Использовать существующий RateLimiter middleware, настроенный строже для `/api/v1/register`.

**Обоснование:**
- Система уже имеет `RateLimiter` service с Redis/in-memory backends
- Предотвращает brute-force enumeration атаки (проверка какие emails существуют)
- Позволяет легитимным пользователям несколько попыток без блокировки аккаунта
- IP-based limiting предотвращает скоординированные атаки

**Конфигурация:**
- Rate limit: 10 request'ов/минута на IP (по-умолчанию, настраивается в `config.py` через переменную `REGISTRATION_RATE_LIMIT`)
- Параметры rate limit полностью configurable per deployment
- Предотвращает credential stuffing и password guessing
- Returns 429 Too Many Requests с Retry-After header'ом

### 6. Аудит логирование: полный захват событий
**Решение:** Логировать все события регистрации (успехи и ошибки) через существующий `AuditService`.

**Обоснование:**
- Видимость безопасности: обнаружение attack patterns
- Compliance: аудит trail для создания аккаунтов
- Troubleshooting: идентификация проблем в flow регистрации

**События для логирования:**
- `REGISTRATION_ATTEMPT_SUCCESS`: email, username, user_id, timestamp
- `REGISTRATION_ATTEMPT_FAILED`: email, username, failure_reason, client_ip, timestamp
- `REGISTRATION_DUPLICATE_EMAIL`: попытанный email, existing user_id
- `REGISTRATION_DUPLICATE_USERNAME`: попытанный username, existing user_id

**Retention policy:**
- Период хранения логов регистрации настраивается через environment переменную `AUDIT_LOG_RETENTION_DAYS`
- По-умолчанию: 90 дней
- GDPR соответствие: failed попытки могут быть удалены после истечения retention периода

### 7. Формат error response: согласуется с существующим API
**Решение:** Использовать Pydantic validation error формат с 422 Unprocessable Entity для validation ошибок.

**Обоснование:**
- FastAPI автоматически валидирует request body и возвращает 422 для validation failures
- Согласуется с существующей обработкой ошибок API
- Клиенты уже ожидают этот формат от других endpoint'ов

**Сценарии ошибок:**
- `400 Bad Request`: Malformed JSON, отсутствуют поля
- `409 Conflict`: Дубликат email или username
- `422 Unprocessable Entity`: Невалидный email формат, слабый пароль
- `429 Too Many Requests`: Rate limit превышен

### 8. Структура response: минимальное представление пользователя
**Решение:** Возвращать новосозданного пользователя (id, email, username, created_at), никогда не возвращать password хеш.

**Обоснование:**
- Предотвращает случайное раскрытие пароля в логах или response'ах
- Достаточно для клиента чтобы подтвердить успешную регистрацию и получить user_id
- Следует принципу наименьших привилегий в данных response'а

### 9. Email подтверждение: опциональное с возможностью отключения
**Решение:** Реализовать email confirmation flow с возможностью отключения через environment переменную.

**Обоснование:**
- Email verification добавляет дополнительный уровень безопасности и подтверждает валидность email
- Environment переменная позволяет использовать разные стратегии для dev/staging/production
- Optionality поддерживает гибкость для разных требований development и production

**Конфигурация:**
- Environment переменная: `REQUIRE_EMAIL_CONFIRMATION` (default: true)
- Если enabled: пользователь создается в статусе `unconfirmed`, требуется confirm по email перед полноценным использованием
- Если disabled: пользователь сразу активен (для development/testing)

### 10. Генерация предложенных usernames при конфликте
**Решение:** При занятом username генерировать и предлагать альтернативные usernames с возможностью отключения.

**Обоснование:**
- Улучшает UX - пользователи получают предложения вместо просто ошибки
- Повышает conversion rate регистрации
- Environment переменная позволяет отключить для случаев когда это не требуется

**Конфигурация:**
- Environment переменная: `SUGGEST_USERNAMES` (default: true)
- Если enabled: возвращать в 409 response список 3-5 доступных вариантов (username + число/символ)
- Если disabled: возвращать только ошибку "Username занят"

**Генерация предложений:**
```
Если запрошен 'john' и занят:
  - john123
  - john_2024
  - john_auth
  - john_service
  - john_v2
```

### 11. Future: Password complexity требования по NIST guidelines
**Решение:** Текущая версия требует только минимальную длину (8 символов). Будущее развитие добавит NIST compliance.

**Обоснование:**
- NIST SP 800-63B рекомендует минимальную длину вместо требований к complexity patterns
- Текущий подход соответствует лучшим практикам
- Можно расширить в будущем без breaking changes

**Future requirements (не в этой версии):**
- Проверка против словарей распространенных паролей
- Проверка против известных breached паролей (Have I Been Pwned)
- Требование неиспользованных ранее паролей при change password

### 12. Ролевая модель: реализация в отдельном изменении
**Решение:** Новые пользователи не получают role assignment. Ролевая модель реализуется позже.

**Обоснование:**
- Ролевая система требует отдельного проектирования и изменения
- Текущее изменение фокусируется на базовой регистрации
- Это может быть добавлено в отдельное изменение после завершения текущего

**Future scope:**
- Система ролей/permissions будет реализована в отдельном change
- Регистрация будет интегрирована с role assignment когда система будет готова

## Риски / Trade-offs

| Риск | Смягчение |
|------|-----------|
| **Timing attack на дублирование** - Attacker может перечислить валидные emails по разности во времени ответа | Всегда выполнять полную валидацию (хеш пароля, построить response) независимо от результата дубликата проверки |
| **Email enumeration атака** - Публичный endpoint раскрывает какие emails зарегистрированы через 409 ошибки | Приемлемо: email как публичный идентификатор стандартно. Если критично - требует email confirmation flow |
| **Подбор пароля в масштабе** - Attacker использует race conditions чтобы обойти rate limit | Rate limit enforced на middleware уровне перед бизнес логикой. Database constraint предотвращает silent failures |
| **Database contention** - Высокий volume регистраций вызывает lock contention на unique constraint | Indexed unique constraints эффективно обрабатывают concurrent inserts. Мониторить performance в high-volume |
| **Слабые пароли в реестре** - Пользователи выбирают пароли ниже требований сложности | Текущий дизайн требует только минимальную длину (8 chars). Будущее - NIST compliance с дополнительными проверками |
| **Потенциал spam регистраций** - Attackers создают множество фейковых аккаунтов | Rate limiting + email confirmation (если enabled) предотвращает масштабный spam |

## План миграции

### Этапы развертывания
1. **Создать новую схему:** Добавить `UserRegister` schema в `app/schemas/user.py`
2. **Реализовать endpoint:** Создать `app/api/v1/register.py` с обработчиком регистрации
3. **Сервис интеграция:** Добавить метод создания пользователя в `UserService` с проверкой дубликатов и логированием
4. **Email confirmation (если enabled):** Реализовать flow для отправки и верификации confirmation email
5. **Username suggestions (если enabled):** Добавить логику генерации alternative usernames при конфликте
6. **Маршрутизация:** Зарегистрировать `/api/v1/register` роутер в `app/main.py`
7. **Конфигурация:** Установить environment переменные в `config.py` (rate limit, email confirmation, suggestions, retention)
8. **Тестирование:** Unit тесты для валидации, integration тесты для endpoint поведения, load testing для rate limiting
9. **Развертывание:** Стандартный процесс, миграции БД не требуются

### Стратегия отката
- **При обнаружении критических проблем:**
  1. Отключить endpoint регистрации удалением роутера из `main.py`
  2. Fallback на seed-based создание пользователей для development
  3. Исследовать проблему в staging перед re-deployment
- **Откат БД не требуется:** User creation - INSERT only, никаких изменений schema
- **Без breaking changes:** Существующие endpoint'ы не затрагиваются

### Шаги верификации
- Ручное тестирование: успешная регистрация, validation ошибки, дублирование, rate limiting
- Email verification (если enabled): проверить отправку и верификацию confirmation email
- Username suggestions (если enabled): проверить генерацию alternative usernames
- Load testing: верифицировать rate limiter предотвращает abuse
- Аудит логирование verification: подтвердить все события залогированы
- Backward compatibility: подтвердить существующие auth flow'ы работают
